# Vercel / AWS / VPS との違い

ここまでで、フロント（Pages）・API（Workers）・データベース（D1）を Cloudflare の無料枠だけで公開
できました。ただ、世の中にはアプリを公開する方法が他にもたくさんあります。AI でアプリを作れるように
なった今、いざ「公開・運用」の段になると「Vercel と何が違うの？」「AWS のほうがいいの？」「サーバーを
借りる（VPS）べき？」と迷うのは自然なことです。

このレクチャーでは **コードを書きません**。代わりに、Cloudflare・Vercel・AWS・VPS（さくらのVPS・
ConoHa など）の 4 つを「無料枠の手厚さ」「公開までの手軽さ」「サーバー管理の有無」という初学者に
とって効いてくる軸で並べて整理します。そのうえで、Cloudflare を **選ぶと良い場面・選ばないほうがいい
場面** を最後にはっきりさせます。

大事なのは「どれが一番偉いか」ではありません。作りたいものによって向き不向きが変わるだけです。
それぞれの得意・不得意を中立に見ていきましょう。

## 学ぶこと

- 各サービスの立ち位置（フロント特化・なんでも屋・サーバー丸ごと・エッジ実行）
- 「サーバー管理がいらない」ことのありがたさと、その裏にある制約
- 従量課金（使った分だけ請求）と固定月額の違い、そして想定外請求のリスク
- Cloudflare の R2 の「egress（下り転送）無料」がなぜ効いてくるのか
- エッジ実行ならではの制約（実行時間の上限・常駐プロセス不可・既存 Node アプリの移植が難しい場合がある）

## 説明

### まず全体像を表で

細かい数値は各社の都合で変わりますが、**性格の違い** はそう簡単には変わりません。2026 年時点の傾向と
して、ざっくり次のように整理できます（無料枠・料金の最新は必ず各社の料金ページを確認してください）。

| | 立ち位置 | 無料枠 | 公開までの手軽さ | サーバー管理 | 課金の形 |
|---|---|---|---|---|---|
| **Cloudflare** | エッジで動くフルスタック基盤 | 手厚い（Workers/Pages/D1/R2） | 手軽（CLI / Git 連携） | 不要 | 無料枠＋従量 |
| **Vercel** | フロント（Next.js 等）特化 | あり（個人向け中心） | とても手軽（Git push で公開） | 不要 | 無料枠＋従量 |
| **AWS** | なんでもできる総合クラウド | 一部に無料利用枠あり | 手間がかかる | 構成しだい（不要〜全部自分） | 細かい従量 |
| **VPS（さくら・ConoHa 等）** | サーバー 1 台を借りる | 基本なし（有料） | 自分でセットアップ | 全部自分 | 固定月額 |

この表の各行を、もう少しかみくだいて見ていきます。

### Vercel: フロントの公開がとにかく速い

Vercel は、React や **Next.js** のようなフロントエンドフレームワークのアプリを公開するのが非常に
得意なサービスです。GitHub にコードを push すると、自動でビルドして公開してくれます。サーバーの
管理は不要です。

特に便利なのが **プレビュー環境** です。プルリクエストごとに「この変更を反映したらこう見える」という
URL が自動で作られるので、本番に出す前に確認できます。チームで開発するときに効きます。

サーバー側の処理が必要なときは、**サーバーレス関数**（リクエストのたびに起動して処理する小さな関数）を
使えます。常時起動のサーバーを持たずに、ちょっとした API を足せるイメージです。

注意したいのは、**無料枠と商用利用の扱い** です。無料プラン（Hobby）は基本的に個人・非商用が前提で、
ビジネス用途では有料プランが必要になります。また、便利な分だけ使い込むと従量で料金が伸びることもある
ので、規約と料金は使い始める前に確認しましょう（最新は Vercel の料金ページを確認してください）。

ひとことで言うと、**フロント中心のアプリを最速で世に出したいなら Vercel は強い** です。

### AWS: なんでもできるが、その分だけ難しい

AWS（Amazon Web Services）は、サーバー・データベース・ストレージ・機械学習まで、**ほぼ何でも揃って
いる総合クラウド** です。大規模なサービスでも組めますし、企業の本番システムでも広く使われています。

その代わり、初学者にとっての壁は **学習コストと運用負荷** です。「サーバーを立てる（EC2）」「保存先を
用意する（S3）」「権限を設定する（IAM）」のように、サービスを組み合わせて自分で構成する必要があり、
覚えることが多くなりがちです。IAM（誰が何をしてよいかの権限管理）はその代表で、最初はとっつきにくい
概念です。

もう一つ気をつけたいのが **従量課金による想定外の請求** です。使った分だけ払う仕組みは合理的ですが、
設定ミスや想定外のアクセスで料金が跳ねることがあります。AWS 自体は予算アラートなどの仕組みを用意して
いますが、初学者が「とりあえず触ってみる」段階では、知らないうちに課金が進むリスクに注意が必要です。

ひとことで言うと、**やりたいことが大きく明確で、運用も含めて手をかけられるなら AWS は頼れる** です。
逆に「まず小さく公開したい」だけなら、最初の一歩としては重く感じやすいです。

### VPS（さくらのVPS・ConoHa など）: サーバーを 1 台借りる

VPS（Virtual Private Server）は、**仮想的なサーバーを 1 台まるごと借りる** サービスです。さくらの
VPS や ConoHa などが有名で、料金は基本的に **固定月額** です（アクセスが少ない月でも多い月でも同じ）。

自由度はとても高く、借りたサーバーに好きな OS を入れ、好きなソフトを動かせます。「常時動かしっぱなしの
プログラム」も置けますし、既存の Node / Express アプリをほぼそのまま動かすこともできます。

その自由の代償が **すべて自分で管理する** ことです。OS のアップデート、ミドルウェア（Web サーバーや
データベース）の導入と設定、そして **セキュリティの更新** まで、自分の責任になります。放置すると
脆弱性を突かれる危険があるため、公開した後も継続して面倒を見る必要があります。

ひとことで言うと、**自由に・常駐で・自分の好きな構成で動かしたいなら VPS は強い**。ただし「サーバー
管理ができる（学ぶ気がある）」ことが前提になります。

### Cloudflare: エッジで動く。無料枠が手厚い

ここまで作ってきた Cloudflare は、**エッジ**（世界中に分散したネットワークの拠点）でコードを動かす
のが特徴です。ユーザーに近い場所で処理されるため速く、そして **サーバー管理は不要** です。OS の更新も
セキュリティパッチも、こちら側で気にする必要がありません。

主な道具立ては次のとおりで、ここまでのハンズオンで一通り触れてきました。

- **Workers** … リクエストを受けて処理を実行する（API・サーバー側ロジック）
- **Pages** … 静的ファイル（HTML/CSS/画像）を配信する（フロント）
- **D1** … SQLite ベースのデータベース
- **R2** … ファイルを置くオブジェクトストレージ

そして初学者にうれしいのが **無料枠の手厚さ** です。とくに **R2 は egress（下り転送）が無料** という
点が効いてきます。画像や動画のように「ダウンロードされる量が多い」データを扱うと、他サービスでは
egress 料金（外に出ていくデータの転送料）がじわじわ効いてくることがありますが、R2 はここが無料なので
読み出しの多い用途で有利になりやすいです（料金体系の最新は Cloudflare の料金ページを確認してください）。

ただし、いいことばかりではありません。エッジ実行ならではの **制約** にも公平に触れておきます。

- **実行時間に上限がある**。1 リクエストでだらだら長時間まわす処理には向きません
- **常駐プロセスを持てない**。「ずっと起動しっぱなしで何かを待つ」タイプのプログラムは置けません
- **既存の Node アプリをそのまま移植しにくいことがある**。Workers は通常の Node.js サーバーとは
  実行環境が異なるため、特定の Node API やライブラリに依存していると、そのままでは動かない場合があります

これらは「Cloudflare がダメ」という話ではなく、**エッジという仕組みの性格** です。性格に合うものを
載せれば強く、合わないものを無理に載せると苦労する、というだけのことです。

### 無料枠・公開の手軽さ・サーバー管理で並べ直す

最後に、初学者が気にする 3 軸でもう一度整理します。

| 観点 | 手軽な順のイメージ | ひとこと |
|---|---|---|
| 無料枠の手厚さ | Cloudflare / Vercel が厚め、AWS は無料利用枠あり、VPS は基本有料 | 小さく始めるなら無料枠の厚いところが安心 |
| 公開までの手軽さ | Vercel・Cloudflare が速い、AWS は構成しだい、VPS はセットアップ必須 | 「URL が出るまで」の早さに差が出る |
| サーバー管理の有無 | Cloudflare・Vercel は不要、AWS は構成しだい、VPS は全部自分 | 管理を持ちたくないなら不要なところを選ぶ |

順位はあくまで「初学者がとっつきやすいか」の目安です。規模が大きくなったり要件が特殊になれば、評価は
変わります。

### どんなときに Cloudflare を選ぶと良いか / 選ばないほうがいいか

選ぶと良い場面の例:

- **小さく速く公開したい**。AI で作ったアプリをまず世に出して反応を見たい
- フロント＋軽い API＋データベースという、**今回作ったような構成** を無料枠中心で動かしたい
- 画像や配布ファイルなど **読み出し（ダウンロード）が多い** データを扱う（R2 の egress 無料が効く）
- **サーバー管理をしたくない**。OS 更新やセキュリティパッチに時間を取られたくない

選ばないほうがいい場面の例:

- **重い常駐バックグラウンド処理** を動かしたい（ずっと起動して待つ・長時間まわす処理）
- 特定クラウドの **専用サービスに強く依存** している（例: そのクラウド固有のマネージドサービス前提）
- **既存の大きな Node / Express アプリをそのまま動かしたい**（移植コストが高くつくことがある）
- 細かく OS やミドルウェアまで **自分で握りたい**（この自由が欲しいなら VPS が向く）

迷ったら、まずは Cloudflare の無料枠で小さく試し、合わなければ別を検討する、という順番が初学者には
おすすめです。今回のハンズオンで作った構成は、その「小さく試す」にちょうどよい形になっています。
