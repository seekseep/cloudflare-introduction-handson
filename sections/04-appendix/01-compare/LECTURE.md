# 公開先の比較（オンプレ/クラウド・VPS/SaaS・ベンダー）

ここまでで、フロント（Pages）・API（Workers）・データベース（D1）を Cloudflare の無料枠だけで公開
できました。ただ、世の中にはアプリを公開する方法が他にもたくさんあります。AI でアプリを作れるように
なった今、いざ「公開・運用」の段になると「Vercel と何が違うの？」「AWS のほうがいいの？」「サーバーを
借りる（VPS）べき？」と迷うのは自然なことです。

このレクチャーは **コードを書きません**。公開先の選び方を **3 段階の絞り込み** として整理します。
いきなり「Cloudflare と AWS どっちがいい？」と比べると混乱しますが、問いには順番があります。大きい
問いから順に答えていくと、自分に合う選択肢が自然としぼられていきます。大事なのは「どれが一番偉いか」
ではありません。作りたいものによって向き不向きが変わるだけです。

## 学ぶこと

- 比較は「並列に全部を比べる」のではなく「大きい問いから順に絞る」と整理しやすいこと
- 第1段：**オンプレ vs クラウド**（ハードを自前で持つか、借りるか）
- 第2段：**VPS vs SaaS**（借りる中でも、サーバーを自分で管理するか、管理を任せるか）
- 第3段：**ベンダー比較**（Vercel / Cloudflare / AWS / GCP / Azure のどれに載せるか）
- Cloudflare を選ぶと良い場面・選ばないほうがいい場面

## 説明

### なぜ「段階」で絞り込むのか

公開先の選択肢は、フラットに並べると数が多すぎて比べきれません。オンプレのサーバールーム、さくらの
VPS、AWS、GCP、Azure、Vercel、Cloudflare……。これらを一度に表に並べても、軸がばらばらで「結局
どれ？」となりがちです。

そこで、**問いの大きい順** に 3 つの段階へ分けます。前の段で答えが出ると、次の段で考える対象がぐっと
減ります。じょうごのように上から絞っていくイメージです。

![3段階の絞り込みをじょうご状に表した図。第1段オンプレ/クラウド、第2段VPS/SaaS、第3段ベンダー選択へと上から下へ選択肢が絞られていく](./images/01-funnel-three-stages.svg)

*図: 公開先の選択を「大きい問いから順に絞る」3段階のじょうご。前の段で答えが出ると次の段の対象が減る。*

| 段階 | 問い | 二択（典型） |
|---|---|---|
| 第1段 | ハードを自前で持つか、借りるか | オンプレミス / クラウド |
| 第2段 | サーバーを自分で管理するか、任せるか | VPS / SaaS・サーバーレス |
| 第3段 | どのベンダーに載せるか | Vercel / Cloudflare / AWS / GCP / Azure |

初学者・小さく始めたい人にとっての「だいたいの結論」を先に置いておきます。第1段はほぼ **クラウド**、
第2段は **管理を任せる側**（サーバーレス／マネージド）、第3段は無料枠で小さく試すなら **Cloudflare** や
**Vercel** から。以下、各段で「何を問われているのか」を順に見ていきます。

:::notice
ここで言う「結論」は、あくまで *初学者が小さく始める* という前提での目安です。会社の本番システムや、
規模・要件が特殊なケースでは、各段の答えは変わります。自分の前提（誰が・何を・どれくらいの規模で）を
意識しながら読み進めてください。
:::

### 第1段：オンプレ vs クラウド

一番大きい問いは「**そもそもサーバー（ハードウェア）を自前で持つのか、借りるのか**」です。

- **オンプレミス（オンプレ）**：機器を **自分（自社）で用意して持つ** 形。社内のサーバールームなどに
  ハードを置き、自分たちで運用します。
- **クラウド**：サーバーを **借りて使う** 形。物理的な機器は事業者（Cloudflare / AWS など）が持ち、
  利用者はネット越しに一部を借ります。電気や水道のように「使う分だけ」使うイメージです。

| 観点 | オンプレミス | クラウド |
|---|---|---|
| 初期費用 | 機器購入で **まとまった出費**（買い切り） | ほぼ不要。**使った分だけ**支払い |
| 立ち上げの速さ | 調達・設置に時間がかかる | アカウントを作れば **すぐ使える** |
| 運用・保守 | 故障対応・電源・空調まで **全部自分** | ハード障害は事業者側が面倒を見る |
| スケール（増減） | 機器を買い足す。**減らしにくい** | 必要に応じて **増減しやすい** |
| 向く規模 | 大規模・長期で安定した負荷 | 小さく始めて様子を見たい用途 |

**初学者にクラウドが向く理由** は、なんといっても初期費用の低さと立ち上げの速さです。今回のハンズオンも、
ハードを 1 台も買わずに無料枠だけで公開できました。「作ってみたけど方向転換したい」というとき契約を
止めるだけで済む **やめやすさ・変えやすさ** も、小さく試す段階と相性がいい性質です。

**それでもオンプレが選ばれる場面** もあります。データを社外・国外に置けない **規制・コンプライアンス**、
すでに自前設備がある **既存資産**、特定ハード前提の **特殊な性能要件**、そして **長期で負荷が一定** な用途。

:::notice
「クラウド＝安い」「オンプレ＝高い」と単純化はできません。**長期で負荷が一定** なら買い切りのオンプレが
総額で有利になることもあります。逆に **負荷が読めない・小さく始めたい** ならクラウドが有利。"使い方
しだいで逆転する" とだけ覚えておけば十分です。
:::

個人や小さく始める用途、今回のような「フロント＋軽い API＋DB」を世に出したいだけなら、答えはほぼ
**クラウド**。上のオンプレ理由に当てはまらない限り、クラウドを選んで次に進んで問題ありません。

### 第2段：VPS vs SaaS

クラウド（借りる）を選ぶと、次は「**同じ借りるでも、サーバーを自分で管理するのか、管理ごと任せるのか**」
です。実はこれは二択ではなく、**管理をどこまで手放すか** の連続的なグラデーションです。

```
自由・自己管理 ◀─────────────────────────────────▶ 手軽・おまかせ

  VPS / IaaS        PaaS・サーバーレス          SaaS
  （サーバーを        （アプリを置く台を          （完成品を
   1台借りる）         借りる）                   使うだけ）

  例: さくらのVPS     例: Cloudflare Workers     例: Gmail,
      AWS EC2            Pages, Vercel              Notion, freee
```

- **VPS / IaaS**：仮想サーバーを 1 台まるごと借り、OS・ミドルウェア・セキュリティ更新まで **全部自分**。
  自由度は最大ですが責任も最大。"常時動かしっぱなしのプログラム" も置けます。
- **PaaS・サーバーレス**：OS やサーバーの管理は事業者に任せ、利用者は **アプリのコードだけ** 載せます。
  今回使った **Cloudflare Workers / Pages** や **Vercel** がここ。手軽な代わりに実行時間の上限など制約あり。
- **SaaS**：自分で作らず、できあがったサービスをそのまま使う（Gmail・Notion・freee など）。グラデー
  ションの右端。

この段は「**どこまでが自分の責任で、どこからが事業者の責任か**」という **責任共有** の境界線として見ると
すっきりします。右に行くほど自分の責任範囲が狭くなります。

| 管理項目 | VPS / IaaS | PaaS・サーバーレス | SaaS |
|---|---|---|---|
| アプリのコード | 自分 | 自分 | 事業者（使うだけ） |
| ミドルウェア（Web サーバー等） | 自分 | 事業者 | 事業者 |
| OS・セキュリティ更新 | 自分 | 事業者 | 事業者 |
| ハードウェア | 事業者 | 事業者 | 事業者 |

:::warning
**「楽＝劣っている」ではありません。** 管理を任せると自分でできることは減りますが、OS のセキュリティ
更新を忘れて脆弱性を突かれる、といった事故のリスクも同時に手放せます。多くの初学者にとっては、自由
より「**事故りにくさ**」のほうが価値が大きいことが多いです。
:::

**自由度と手軽さのトレードオフ** で言うと、既存の Node / Express をそのまま動かしたい・常駐プロセスを
持ちたい・OS まで握りたい（そして管理を学ぶこと自体が目的）なら **VPS**。とにかく早く楽に公開したい・
サーバー管理に時間を取られたくないなら **サーバーレス**。サーバー管理を学ぶこと自体が目的でなければ、
**管理を任せる側** から始めるのが楽です。今回サーバーを一切管理せず公開できたのは、この側を選んで
いたからです。

### 第3段：ベンダー比較（Vercel / Cloudflare / AWS / GCP / Azure）

管理を任せると決めたら、最後は「**では、どのベンダーに載せるのか**」です。細かい数値は各社の都合で
変わりますが、**性格の違い** はそう簡単には変わりません（無料枠・料金の最新は必ず各社の料金ページを確認）。

| | 立ち位置 | 無料枠 | 公開までの手軽さ | 得意分野 |
|---|---|---|---|---|
| **Cloudflare** | エッジで動くフルスタック基盤 | 手厚い（Workers/Pages/D1/R2） | 手軽（CLI / Git 連携） | エッジ実行・配信・低コスト |
| **Vercel** | フロント（Next.js 等）特化 | あり（個人向け中心） | とても手軽（Git push で公開） | フロントの公開・プレビュー |
| **AWS** | なんでもできる総合クラウド | 一部に無料利用枠あり | 手間がかかる | 規模・サービスの幅 |
| **GCP** | 総合クラウド（データ・AI に強い） | 一部に無料利用枠あり | 手間がかかる | データ分析・機械学習 |
| **Azure** | 総合クラウド（企業・MS 連携） | 一部に無料利用枠あり | 手間がかかる | 企業 IT・Microsoft 連携 |

- **Vercel**：React や **Next.js** のフロントを公開するのが非常に得意。GitHub に push すると自動ビルド・
  公開。PR ごとの **プレビュー環境** が便利。ただし無料プラン（Hobby）は個人・非商用が前提で、商用は
  有料。**フロント中心のアプリを最速で世に出したいなら強い**。
- **Cloudflare**：**エッジ**（世界中に分散した拠点）でコードを動かすのが特徴。ユーザーに近い場所で処理
  され速く、サーバー管理は不要。道具立ては Workers（API）・Pages（フロント）・D1（DB）・R2（ストレージ）。
  初学者にうれしいのが **無料枠の手厚さ** で、とくに **R2 は egress（下り転送）が無料**。画像や動画のように
  ダウンロード量が多いデータは他社だと egress 料金が効いてきますが、R2 はここが無料で読み出しの多い
  用途に有利です。
- **AWS / GCP / Azure**：サーバー・DB・ストレージ・機械学習までほぼ何でも揃う **総合クラウド三強**。
  大規模でも組め、企業の本番でも広く使われます。共通の壁は **学習コストと運用負荷**（特に権限管理）、
  そして **従量課金による想定外の請求** です。

Cloudflare のエッジ実行には **制約** もあります。フェアに挙げておきます。

- **実行時間に上限がある**。1 リクエストでだらだら長時間まわす処理には向かない
- **常駐プロセスを持てない**。ずっと起動して何かを待つタイプは置けない
- **既存の Node アプリをそのまま移植しにくいことがある**。実行環境が通常の Node.js と異なるため、
  特定の Node API やライブラリに依存していると動かない場合がある

これらは「Cloudflare がダメ」ではなく **エッジという仕組みの性格** です。合うものを載せれば強く、合わない
ものを無理に載せると苦労する、というだけのことです。

:::warning
総合クラウド（AWS / GCP / Azure）は **想定外の請求** に特に注意してください。無料利用枠を超えたり、
消し忘れたリソースが動き続けたりすると、月末に高額な請求が来ることがあります。学習で触るときは、
**必ず予算アラートを設定** し、使い終わったリソースは消す習慣をつけましょう。
:::

### どんなときに Cloudflare を選ぶ／選ばない

**選ぶと良い場面**：

- **小さく速く公開したい**。AI で作ったアプリをまず世に出して反応を見たい
- フロント＋軽い API＋DB という、**今回作ったような構成** を無料枠中心で動かしたい
- 画像や配布ファイルなど **読み出し（ダウンロード）が多い** データを扱う（R2 の egress 無料が効く）
- **サーバー管理をしたくない**。OS 更新やセキュリティパッチに時間を取られたくない

**選ばないほうがいい場面**：

- **重い常駐バックグラウンド処理** を動かしたい（ずっと起動して待つ・長時間まわす処理）
- 特定クラウドの **専用サービスに強く依存** している
- **既存の大きな Node / Express アプリをそのまま動かしたい**（移植コストが高くつくことがある）
- 細かく OS やミドルウェアまで **自分で握りたい**（この自由が欲しいなら VPS が向く）

迷ったら、まずは Cloudflare の無料枠で小さく試し、合わなければ別を検討する、という順番が初学者には
おすすめです。今回のハンズオンで作った構成（完成形は
[D1 でデータを保存する](../../03-build-app/02-d1/LECTURE.md) で確認できます）は、その「小さく試す」に
ちょうどよい形になっています。なお、どのサービスを選んでも API キーやパスワードの扱いを間違えると
事故につながります。安全な公開の基礎は
[秘匿情報の扱い方](../../02-security/01-basic/LECTURE.md) を参照してください。
