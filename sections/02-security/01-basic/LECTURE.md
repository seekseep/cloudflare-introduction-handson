# セキュリティの3要素（可用性・機密性・完全性）

![セキュリティの3要素](./images/00-thumbnail.svg)

インターネットに公開したアプリは、**世界中からアクセスされる可能性があります**。
その中には秘密情報（シークレット）を盗む人、アプリを悪用する人、大量アクセスでサービスを止める人もいます。

![攻撃者があなたのサーバーを攻撃する図。攻撃の目的（データの窃取・サービス停止・Webサイトの改ざん・金銭の要求）を列挙](./images/00-attack-purposes.svg)

*図: 公開したアプリには攻撃者も到達できる。目的はデータの窃取・サービス停止・改ざん・金銭要求などさまざま。*

## セキュリティの3要素

守るべきものを次の3つの性質に分けて考えます。頭文字を取って **CIA** と呼びます。

| 要素 | 英語 | ひとことで言うと | 崩れると何が起きるか |
| --- | --- | --- | --- |
| **機密性** | Confidentiality | 見てよい人だけが見られる | 秘密の値・個人情報が漏れる |
| **完全性** | Integrity | 中身が正しく保たれる | データが書き換えられる・踏み台にされる |
| **可用性** | Availability | 使いたいときに使える | 必要なときにサービスを利用できない |

**どれか1つだけでは足りません。** 秘密を守っても（機密性）データを書き換えられたら（完全性）意味がなく、そもそも止まっていたら（可用性）使えません。

![機密性・完全性・可用性の3要素が中央のアプリを三方から支える CIA トライアングル](./images/05-cia-triad.svg)

*図: 「機密性・完全性・可用性」の3点で支える。どれか1つが欠けても守れない。*

## 機密性

![機密性のイメージ。中央に鍵のかかった「機密データ（APIキー・個人情報）」の箱。左の「見てよい人（鍵を持つ）」は箱を開けて見られる（✅）。右の「見てはいけない人（鍵がない）」は境界で止められ入れない（⛔）](./images/09-confidentiality.svg)

**見てよい人だけが見られる**ことです。守る対象は、API キーや DB パスワードなどの**秘密情報（シークレット）**と、氏名・メールなどの**個人情報**です。

- **被害**：API キーが漏れると、なりすましによる**不正利用や高額請求**につながる。個人情報が漏れれば利用者に直接被害が及び、**信用失墜・損害賠償**にもなる。
- **対策**：**必要な人・プログラムだけがアクセスできる場所で管理する**。①シークレットはフロントに置かずサーバー側で使う → ②Git に入れない（`.gitignore`）→ ③環境変数、本番はシークレット管理機能で扱う。個人情報は必要以上に集めない・返さない、パスワードはハッシュ化・通信は HTTPS。

:::danger
「画面に出していない」＝「隠せている」ではありません。フロントに書いた値は表示していなくてもブラウザに届いています。万一 Git に上げてしまった鍵は、発行元で無効化します。
:::

## 完全性

![完全性のイメージ。中央に盾で守られた「データ（投稿・残高・在庫）」の箱。左の「権限のある人（本人・管理者）」は書き換えOK（✅）。右の「権限のない人（他人）」は境界で止められ書き換えNG（⛔）](./images/10-integrity.svg)

**中身が正しく保たれる**ことです。許可されていない変更・削除・破壊を防ぎ、正しい状態を維持します。**悪意ある攻撃だけでなく、操作ミスやシステム障害でデータが壊れること**も含みます。

- **被害**：出品価格や他人の投稿を勝手に書き換えられる、残高・在庫・注文数などの値がいじられる。「誰が書いたか分からない」「戻せない」状態だと、気づくことも復旧もできない。
- **対策**：本人（や管理者）だけが書き換え・削除できるよう**アクセス権限を設定**し、それを**サーバー側で検証**する（フロントのチェックは回避される）。誰が何を変えたか**ログに残し**、壊れても戻せるよう**バックアップ**する。詳しくは [Workers で API を動かす](../../03-build-app/01-workers/LECTURE.md)・[D1 でデータを保存する](../../03-build-app/02-d1/LECTURE.md) で扱います。

## 可用性

![可用性のイメージ。中央に動き続ける「サービス（サーバー・データ／止まらない・戻せる）」の箱。左の「使いたい人（いつでも）」は使える（✅）。右の「障害・攻撃（ダウン・財布攻撃）」はサービスを止められない（⛔）](./images/11-availability.svg)

**使いたいときに使える**ことです。公開して終わりではなく、安定して動き続けることが目的です。

- **被害**：攻撃・障害・改修ミスでサービスが止まる。とくに外部 API の従量課金は、大量リクエスト（財布攻撃）で**利用料金が一気に膨らむ**ことがある。
- **対策**：**戻せる・気づける・止めない・起こさない**を用意する。定期**バックアップ**（＝完全性とも共通）、ログや死活監視での**監視**、複数台で動かす**冗長化**（フェイルオーバー・負荷分散）、利用上限・予算アラートでの**予防**。

## トレードオフする３要素

3要素はすべてを同時に最大化できるとは限らず、実際のシステムでは**片方を上げると別の片方が下がる**トレードオフになることがよくあります。天秤のように、一方に重みをかけると反対側が持ち上がってしまう関係です。

ただし、これは**必ずそうなるわけではありません**。ECC メモリやチェックサムのように、可用性をほとんど下げずに完全性を高められる仕組みや、CDN・負荷分散のように機密性を下げずに可用性を高められる仕組みもあります。ここで紹介するのは「ある要素を極端に優先する設計では、別の要素が犠牲になりやすい」という**代表的な場面**です。3つ見てみましょう。

### 機密性を上げると → 可用性が下がる

機密性を高める一番の方法は「見られる人・使える人を絞る」ことです。たとえばログインに多要素認証（MFA）を必須にすれば、パスワードが漏れても不正アクセスを防げます。しかしその分、認証サーバーが障害を起こすと正規の利用者までログインできなくなり、手続きも増えて「気軽に使えない」サービスになります。社内システムで USB メモリや社外からのアクセスを一律禁止するのも同じで、情報漏えいは防げても日々の業務効率は確実に落ちます。守りを固めるほど、使いやすさ（可用性）は削られていきます。

![天秤の左に機密性、右に可用性。機密性を強くすると可用性側が下がる図](./images/06-tradeoff-conf-avail.svg)

*図: 機密性を強くするほど、使いやすさ（可用性）は下がる。*

### 完全性を上げると → 可用性が下がる

**厳密な整合性を最優先する設計では**、「正しい状態を厳密に守る」仕組みが処理の重さや待ち時間と引き換えになることがあります。たとえばデータベースで厳密なトランザクションやロックをかければ、同時更新による矛盾は防げますが、順番待ちが増えて性能は落ちます。掲示板で「投稿はすべて運営が1件ずつ承認してから公開」にすれば、誤った内容や荒らしは確実に止められますが、投稿してもすぐには表示されず、リアルタイムに使えるという可用性は損なわれます。正確さを突き詰めるほど、すぐ使えるという快適さは犠牲になります。

![天秤の左に完全性、右に可用性。完全性を強くすると可用性側が下がる図](./images/07-tradeoff-integ-avail.svg)

*図: 完全性を強くするほど、すぐ使える快適さ（可用性）は下がる。*

### 可用性を上げると → 機密性・完全性が下がる

**可用性を優先する設計では**、やり方によっては機密性や完全性が緩むことがあります。たとえば認証なしで誰でも叩ける公開 API を用意すれば使い勝手は最高ですが、その分だけ情報漏えいのリスクは高まります（機密性↓）。また、障害時でもサービスを止めないようにキャッシュやレプリカ（コピー）を優先して返すと、アクセスは途切れませんが、一時的に古い・食い違ったデータが表示されることがあります（完全性↓）。もっとも、CDN や負荷分散のように**可用性を上げても機密性を下げない**手段もあります。緩むかどうかは選ぶやり方しだいです。

![天秤の左に可用性、右に機密性・完全性。可用性を強くすると機密性・完全性側が下がる図](./images/08-tradeoff-avail-conf.svg)

*図: 可用性を強くするほど、秘密（機密性）や正確さ（完全性）は下がる。*

このように、どれか1つを100点にしようとすると別の要素が犠牲になります。だから現実の設計は「全部100点」ではなく、**そのアプリで何を一番守りたいかを決めて、バランスを取る**ことになります。

たとえば――

- **気軽な雑談掲示板** なら、多少荒れても使いやすさが大事なので **可用性を重め** に。
- **氏名や住所を扱う会員サービス** なら、漏れたら取り返しがつかないので **機密性を重め** に。
- **お金や在庫を扱うアプリ** なら、数字を1円でも狂わせたくないので **完全性を重め** に。

「このアプリでは何が一番痛いか」を先に決めるのが、セキュリティ設計の出発点です。
