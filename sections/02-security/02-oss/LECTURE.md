# OSS のセキュリティ

このハンズオンでも、`npm install` を打つたびに大量のパッケージがダウンロードされてきました。
あれは**他人が書いたコード（OSS）** です。自分のアプリは、自分で書いた部分よりも、はるかに多くの「他人のコード」の上に乗っています。

便利さの裏側で、その「他人のコード」が **攻撃の入り口** になる事故が、近年くり返し起きています。
このレクチャーではOSS とサプライチェーンという考え方を押さえていきましょう。

## 学ぶこと

- OSS（オープンソースソフトウェア）とは何か、なぜセキュリティ上の問題になりうるのか
- 「依存関係」と「サプライチェーン」という考え方 ―― 自分のコードより **他人のコードの方が多い** こと
- 実際に起きた 3 つの事件（left-pad / npm 乗っ取り / Next.js 認証バイパス）から何が学べるか
- lockfile・`npm ci`・`npm audit`・Dependabot といった、今日から使える最低限の守り

## 説明

### OSS とは / なぜセキュリティが問題になるのか

**OSS（オープンソースソフトウェア）** は、ソースコードが公開され、だれでも使える形で配布されているソフトウェアです。
npm で入ってくるパッケージの大半がこれにあたります。
世界中の開発者が作った部品を無料で再利用できるので、私たちは車輪の再発明をせずに、短時間でアプリを組み立てられます。

問題は、その部品を **「自分では中身を確認していない」まま使っている** ことです。

- 誰が書いたのか、どんな更新が入ったのかを、いちいち読んではいない
- インストール時にパッケージが実行するスクリプト（後述の `postinstall` など）を把握していない
- 部品がさらに別の部品を呼んでいて、全体像が見えない

つまり OSS のセキュリティ問題の根っこは、**「信頼して取り込んでいる他人のコードが、いつのまにか悪意あるコードに入れ替わる（あるいは壊れる）」** という一点に集約されます。

### 依存関係とサプライチェーンという考え方

自分が `package.json` に書いたパッケージを **直接依存（direct dependencies）** と呼びます。ところが、
その直接依存もまた別のパッケージに依存していて、そのまた先も…… と枝分かれしていきます。
この「依存の依存」を **推移的依存関係（transitive dependencies）** と呼びます。

[Webエンジニアなら知っておきたい、依存関係の分類のお話](https://engineers.fenrir-inc.com/entry/2024/08/20/133445#%E6%8E%A8%E7%A7%BB%E7%9A%84%E4%BE%9D%E5%AD%98%E9%96%A2%E4%BF%82Transitive-dependencies)

![自分のアプリから直接依存・その依存・さらにその依存へと左から右にツリー状に広がる依存関係の図](./images/01-dependency-iceberg.svg)

<!-- genfig: 依存関係が左から右へ広がるツリー（PATH/BRANCHING スキーマ）図。左端に「自分のアプリ（自分で書いた分）」を1つ置き、右へ行くほど枝分かれして数が増える。列ごとに色を分ける: 自分のアプリ=青 / 直接依存（framework・router・validator など数個）=ティール / その依存=紫 / さらにその依存…=オレンジ。曲線で親子を結び、右へ行くほどノード数が指数的に増える様子を描く。伝えたいこと=「直接書いた依存の先に、自分では選んでいない他人のコードがツリー状にどんどん広がっている」。 -->
*図: 依存関係は左から右へツリー状に広がる。右へ行くほど、自分では選んでいない他人のコードが指数的に増えていく。*

現代の JavaScript プロジェクトでは、`package.json` に 10 個ほどしか書いていなくても、`node_modules`には数百〜千を超えるパッケージが展開される、というのはごく普通のことです。**あなたのアプリが実際に動かしているコードの大半は、あなたが直接選んでいない他人のコード** です。

この「部品が部品を呼び、それらを取り寄せて最終製品を組み立てる」構造を、製造業になぞらえて[**ソフトウェアサプライチェーン（供給網）**](https://www.sompocybersecurity.com/column/glossary/software-supply-chain) と呼びます。

そして、この供給網のどこか一箇所に悪意あるコードや欠陥が混じると、それを取り込んだすべてのアプリに影響が及ぶ攻撃を**サプライチェーン攻撃** と呼びます。

さらに厄介なのが **`postinstall`（インストール時スクリプト）** です。npm のパッケージは、`npm install` した瞬間に、あらかじめ仕込まれたスクリプトを **あなたのマシン上で自動実行** できます。本来は「ネイティブ拡張のビルド」などのための仕組みですが、悪意ある側から見れば「インストールさせるだけでコードを走らせられる」入り口になります。後述の事件でも、この仕組みが悪用されました。

この危険性への対策として、pnpm はインストール時のビルドスクリプトを **デフォルトでブロック** し、あなたが明示的に承認したパッケージだけ実行を許可します（`pnpm approve-builds`）。詳しくは [pnpm approve-builds（公式ドキュメント）](https://pnpm.io/ja/cli/approve-builds) を参照してください。

:::note
サプライチェーン攻撃が怖いのは、**あなた自身のコードに一行のバグもなくても被害を受ける** 点です。
守るべき対象が「自分のコード」から「自分が取り込んでいる他人のコード全部」へと広がります。
:::

### ニュース①：left-pad 事件（2016）

最初は、悪意ではなく **たった 1 つのパッケージが消えただけ** で連鎖崩壊が起きた古典的な事件です。

- **時期**：2016 年 3 月 22 日
- **何が起きたか**：`left-pad` という、文字列の左側を指定文字で埋めるだけの **わずか 11 行** のパッケージが、作者によって npm から公開停止（unpublish）された
- **影響**：`left-pad` は Babel（トランスパイラ）や React、webpack など無数のプロジェクトが **推移的依存関係** として引いていたため、世界中で「依存が見つからずビルドできない」障害が連鎖した。報道によれば当時 1,500 万回以上ダウンロードされていた
- **背景**：作者が別のパッケージ名をめぐる npm 社とのトラブルから、自分の全パッケージを取り下げた結果だったとされる
- **その後**：数時間後に npm 社が `left-pad` を手動で復元。さらに npm は **「公開から 24 時間以上経過し、かつ他のパッケージが依存しているものは unpublish できない」** ようポリシーを変更した

学びは明快です。**たった 11 行の、しかも自分では選んでいない間接依存が消えるだけで、巨大なサービス群が止まる。** これは悪意すら必要としない、依存の深さそのものが持つ脆さの実例です。

参考:

- [npm 公式ブログ「kik, left-pad, and npm」](https://blog.npmjs.org/post/141577284765/kik-left-pad-and-npm)
- [The Register の報道（2016）](https://www.theregister.com/2016/03/23/npm_left_pad_chaos/)
- [Wikipedia: npm left-pad incident](https://en.wikipedia.org/wiki/Npm_left-pad_incident)

### ニュース②：npm 人気パッケージのメンテナ乗っ取り（2025）

left-pad が「消えて壊れた」事件なら、こちらは **悪意あるコードを混ぜ込まれた** 事件です。しかも
狙われたのが、ほぼすべての JS プロジェクトが引いているような超人気パッケージでした。

- **時期**：2025 年 9 月 8 日
- **手口**：`chalk` `debug` などを管理するメンテナ（Qix 氏）が、**`support@npmjs.help` を名乗る偽の 2 要素認証リセット通知メール（フィッシング）** に引っかかり、アカウントを乗っ取られた
- **何が起きたか**：`chalk` `debug` `ansi-styles` `strip-ansi` `color-convert` `wrap-ansi` など 18 前後の高頻度パッケージに、悪意あるバージョンが公開された。これらは合計で **週あたり 20 億回超** ダウンロードされる規模だった
- **仕込まれた中身**：報道によれば、ブラウザ上で暗号資産（web3）の送金・承認フローに割り込み、送金先アドレスを攻撃者のものへ **こっそり書き換える** マルウェアだった。画面表示は正常に見せかける
- **収束**：公開から数時間のうちにメンテナ本人が乗っ取りを公表し、悪意あるバージョンの削除が進んだ

さらに深刻だったのが、同じ 9 月に確認された **自己増殖型ワーム「Shai-Hulud」** です。報道によれば、これは感染したパッケージが **開発者の認証情報を盗み、その開発者が管理する別のパッケージに自分自身をコピーして再公開する** ことで、人手を介さずに感染を広げました。最初の 24 時間で約 200、その後の集計では 500 を超えるパッケージが影響を受けたとされ、11 月には亜種（Shai-Hulud 2.0）も報告されています。CISA（米国のサイバーセキュリティ機関）も npm エコシステム全体への警告を出しました。

:::danger
この事件の教訓は 2 つあります。ひとつは **「有名で広く使われている ＝ 安全」ではない** こと。むしろ広く使われているからこそ狙われます。
もうひとつは、**きっかけがメンテナ 1 人へのフィッシングメール**だったこと。
人気パッケージの管理者アカウントが 1 つ乗っ取られるだけで、それに依存する無数のアプリが一斉に危険にさらされます。
:::

参考:

- [Socket「npm author Qix compromised」](https://socket.dev/blog/npm-author-qix-compromised-in-major-supply-chain-attack)
- [The Hacker News（週20億DL規模の解説）](https://thehackernews.com/2025/09/20-popular-npm-packages-with-2-billion.html)
- [Wiz Blog（影響範囲の分析）](https://www.wiz.io/blog/widespread-npm-supply-chain-attack-breaking-down-impact-scope-across-debug-chalk)

### ニュース③：Next.js の認証バイパス（CVE-2025-29927 / 2025）

3 つ目は、悪意ある第三者ではなく、**広く使われているフレームワーク自身の設計上の欠陥** が
セキュリティホールになった例です。

- **識別子 / 深刻度**：CVE-2025-29927。CVSS 9.1（Critical）
- **対象**：人気の JS フレームワーク Next.js の **ミドルウェア** による認可（アクセス制御）
- **仕組み**：Next.js は内部処理で `x-middleware-subrequest` というヘッダを使い、ミドルウェアの無限ループを防いでいた。ところがこのヘッダを **外部から付けて送る** と、ミドルウェアの実行そのものがスキップされてしまう。ミドルウェアで「ログインしているか」を確認していた場合、その **認可チェックを丸ごと迂回** できてしまった
- **影響バージョンと修正版**（報道・NVD による）：
  - 12.x 系：`12.3.5` より前が影響（修正版 `12.3.5`）
  - 13.x 系：`13.5.9` より前が影響（修正版 `13.5.9`）
  - 14.x 系：`14.2.25` より前が影響（修正版 `14.2.25`）
  - 15.x 系：`15.2.3` より前が影響（修正版 `15.2.3`）
- **暫定対策**：修正版へ上げるのが本筋。すぐに上げられない場合は、Web サーバやプロキシの側で外部から来た `x-middleware-subrequest` ヘッダを **落とす（拒否する）** ことが緩和策として案内された

ここでの学びは、**「フレームワークを使っているから安全」ではなく、そのフレームワーク自体を最新の安全なバージョンに保つのも自分の責任** だということです。認可のような重要な処理を、フレームワークの 1 つの仕組み（ここではミドルウェア）だけに頼りきると、その仕組みが破られたとき無防備になります。

参考:

- [NVD: CVE-2025-29927](https://nvd.nist.gov/vuln/detail/CVE-2025-29927)
- [JFrog による解説](https://jfrog.com/blog/cve-2025-29927-next-js-authorization-bypass/)
- [Datadog Security Labs の技術分析](https://securitylabs.datadoghq.com/articles/nextjs-middleware-auth-bypass/)

### ニュースから学べること

3 つの事件は、それぞれ「消える」「乗っ取られる」「設計に穴があった」と原因は違いますが、対策は
かなり共通しています。

- **更新は「すぐ・むやみに」ではなく「見極めて」上げる**
  - セキュリティ修正は速やかに取り込む。一方で、公開直後の最新版に飛びつくのは避け、少し様子を見る
    （②のように公開直後の版が汚染されていることがある）。「古すぎ」も「新しすぎ」もリスク、という
    バランス感覚を持つ
- **lockfile を必ずコミットし、`npm ci` で入れる**
  - `package-lock.json` は「実際に入れた各パッケージの正確なバージョンと中身のハッシュ」を記録する。
    これをコミットしておけば、次に入れるときも **同じ中身** が保証される
  - CI や本番では `npm install` ではなく **`npm ci`** を使う。`npm ci` は lockfile に厳密に従い、
    lockfile と `package.json` が食い違えば失敗するので、意図しないバージョンの混入を防げる
- **`npm audit` と Dependabot で「穴の空いた依存」を検知する**
  - `npm audit` は、いま入っている依存に既知の脆弱性がないかを一覧してくれる
    ```bash
    npm audit
    ```
  - GitHub の **Dependabot** を有効にすると、依存に既知の脆弱性が見つかったときに警告し、更新用の
    Pull Request を自動で出してくれる。③のようなフレームワークの CVE も、これで気づける
- **そもそも依存を増やしすぎない**
  - 「11 行のためにパッケージを 1 つ増やす」より、自分で数行書いた方が安全なこともある（①の教訓）。
    依存を 1 つ足すたびに、その先の間接依存も、`postinstall` も、まるごと信頼していることを意識する

### まず最低限やること

全部を完璧にやる必要はありません。個人開発や小さなチームなら、まずは次の 4 つで十分に足腰が強く
なります。

1. **`package-lock.json` を必ず git にコミットする**（消したり `.gitignore` に入れたりしない）
2. **CI・本番では `npm ci` を使う**（`npm install` で勝手にバージョンが動くのを防ぐ）
3. **GitHub の Dependabot を有効にする**（脆弱性の通知と更新 PR を自動化）
4. **依存を足すとき一呼吸おく**（本当に必要か・週の使われ方・最終更新日をざっと見る）

OSS は、私たちが小さな労力で大きなものを作れる理由そのものです。だからこそ「便利だけど、他人の
コードに乗っている」という前提を忘れず、**最新に保つ・固定する・監視する** の 3 つを習慣にして
おきましょう。
