# ベンダー比較（Vercel / Cloudflare / AWS / GCP / Azure）

3 段階の最後、第3段の問いです。[オンプレ vs クラウド](../02-onpremise-vs-cloud/LECTURE.md) で
「借りる」を、[VPS vs SaaS](../03-vps-vs-saas/LECTURE.md) で「管理を任せる」を選びました。残るは
**では、どのベンダーに載せるのか** です。

ここでは、管理を任せて使える主要な公開先 — **Vercel・Cloudflare・AWS・GCP・Azure** の 5 つを、
初学者にとって効いてくる軸（無料枠・公開の手軽さ・得意分野）で並べて整理します。そのうえで、
Cloudflare を **選ぶと良い場面・選ばないほうがいい場面** をはっきりさせます。

くり返しになりますが、「どれが一番偉いか」ではありません。作りたいものによって向き不向きが変わる
だけです。それぞれの得意・不得意を中立に見ていきましょう。

## 学ぶこと

- 各ベンダーの立ち位置（フロント特化・エッジ実行・総合クラウド三強）
- 従量課金（使った分だけ請求）の便利さと、想定外請求のリスク
- Cloudflare の R2 の「egress（下り転送）無料」がなぜ効いてくるのか
- エッジ実行ならではの制約（実行時間の上限・常駐プロセス不可・既存 Node アプリの移植）

## 説明

### まず全体像を表で

細かい数値は各社の都合で変わりますが、**性格の違い** はそう簡単には変わりません。2026 年時点の傾向と
して、ざっくり次のように整理できます（無料枠・料金の最新は必ず各社の料金ページを確認してください）。

| | 立ち位置 | 無料枠 | 公開までの手軽さ | 得意分野 |
|---|---|---|---|---|
| **Cloudflare** | エッジで動くフルスタック基盤 | 手厚い（Workers/Pages/D1/R2） | 手軽（CLI / Git 連携） | エッジ実行・配信・低コスト |
| **Vercel** | フロント（Next.js 等）特化 | あり（個人向け中心） | とても手軽（Git push で公開） | フロントの公開・プレビュー |
| **AWS** | なんでもできる総合クラウド | 一部に無料利用枠あり | 手間がかかる | 規模・サービスの幅 |
| **GCP** | 総合クラウド（データ・AI に強い） | 一部に無料利用枠あり | 手間がかかる | データ分析・機械学習 |
| **Azure** | 総合クラウド（企業・MS 連携） | 一部に無料利用枠あり | 手間がかかる | 企業 IT・Microsoft 連携 |

### Vercel: フロントの公開がとにかく速い

Vercel は、React や **Next.js** のようなフロントエンドフレームワークのアプリを公開するのが非常に
得意なサービスです。GitHub にコードを push すると、自動でビルドして公開してくれます。サーバーの
管理は不要です。

特に便利なのが **プレビュー環境** です。プルリクエストごとに「この変更を反映したらこう見える」という
URL が自動で作られるので、本番に出す前に確認できます。チームで開発するときに効きます。

サーバー側の処理が必要なときは、**サーバーレス関数**（リクエストのたびに起動して処理する小さな関数）を
使えます。常時起動のサーバーを持たずに、ちょっとした API を足せるイメージです。

注意したいのは、**無料枠と商用利用の扱い** です。無料プラン（Hobby）は基本的に個人・非商用が前提で、
ビジネス用途では有料プランが必要になります。また、便利な分だけ使い込むと従量で料金が伸びることもある
ので、規約と料金は使い始める前に確認しましょう（最新は Vercel の料金ページを確認してください）。

ひとことで言うと、**フロント中心のアプリを最速で世に出したいなら Vercel は強い** です。

### Cloudflare: エッジで動く。無料枠が手厚い

ここまで作ってきた Cloudflare は、**エッジ**（世界中に分散したネットワークの拠点）でコードを動かす
のが特徴です。ユーザーに近い場所で処理されるため速く、そして **サーバー管理は不要** です。OS の更新も
セキュリティパッチも、こちら側で気にする必要がありません。

主な道具立ては次のとおりで、ここまでのハンズオンで一通り触れてきました。

- **Workers** … リクエストを受けて処理を実行する（API・サーバー側ロジック）
- **Pages** … 静的ファイル（HTML/CSS/画像）を配信する（フロント）
- **D1** … SQLite ベースのデータベース
- **R2** … ファイルを置くオブジェクトストレージ

そして初学者にうれしいのが **無料枠の手厚さ** です。とくに **R2 は egress（下り転送）が無料** という
点が効いてきます。画像や動画のように「ダウンロードされる量が多い」データを扱うと、他サービスでは
egress 料金（外に出ていくデータの転送料）がじわじわ効いてくることがありますが、R2 はここが無料なので
読み出しの多い用途で有利になりやすいです（料金体系の最新は Cloudflare の料金ページを確認してください）。

ただし、いいことばかりではありません。エッジ実行ならではの **制約** にも公平に触れておきます。

- **実行時間に上限がある**。1 リクエストでだらだら長時間まわす処理には向きません
- **常駐プロセスを持てない**。「ずっと起動しっぱなしで何かを待つ」タイプのプログラムは置けません
- **既存の Node アプリをそのまま移植しにくいことがある**。Workers は通常の Node.js サーバーとは
  実行環境が異なるため、特定の Node API やライブラリに依存していると、そのままでは動かない場合があります

これらは「Cloudflare がダメ」という話ではなく、**エッジという仕組みの性格** です。性格に合うものを
載せれば強く、合わないものを無理に載せると苦労する、というだけのことです。

### AWS / GCP / Azure: 総合クラウド三強

AWS（Amazon）・GCP（Google）・Azure（Microsoft）は、サーバー・データベース・ストレージ・機械学習まで、
**ほぼ何でも揃っている総合クラウド** です。大規模なサービスでも組めますし、企業の本番システムでも
広く使われています。この 3 つは "三大クラウド" と呼ばれ、できることの幅という点では横並びです。

性格の違いをざっくり言うと、次のような傾向があります（あくまで傾向で、実際はどれも何でもできます）。

- **AWS** … 最も歴史が長く、サービスの数と情報量が豊富。迷ったら情報が見つけやすい
- **GCP** … データ分析や機械学習まわりに強いと言われる。Google のインフラ系と相性がよい
- **Azure** … 企業 IT や Microsoft 製品（Microsoft 365 など）との連携に強い。会社で使うケースが多い

共通する初学者にとっての壁は **学習コストと運用負荷** です。「サーバーを立てる」「保存先を用意する」
「権限を設定する」のように、サービスを組み合わせて自分で構成する必要があり、覚えることが多くなりがちです。
特に **権限管理**（誰が何をしてよいかの設定）は、どのクラウドでも最初はとっつきにくい概念です。

もう一つ気をつけたいのが **従量課金による想定外の請求** です。使った分だけ払う仕組みは合理的ですが、
設定ミスや想定外のアクセスで料金が跳ねることがあります。各社とも予算アラートなどの仕組みは用意して
いますが、初学者が「とりあえず触ってみる」段階では、知らないうちに課金が進むリスクに注意が必要です。

:::warning
総合クラウド（AWS / GCP / Azure）は **想定外の請求** に特に注意してください。無料利用枠を超えたり、
消し忘れたリソースが動き続けたりすると、月末に高額な請求が来ることがあります。学習で触るときは、
**必ず予算アラートを設定** し、使い終わったリソースは消す習慣をつけましょう。
:::

ひとことで言うと、**やりたいことが大きく明確で、運用も含めて手をかけられるなら総合クラウドは頼れる**
です。逆に「まず小さく公開したい」だけなら、最初の一歩としては重く感じやすいです。

### 無料枠・公開の手軽さで並べ直す

最後に、初学者が気にする観点でもう一度整理します。

| 観点 | 手軽な順のイメージ | ひとこと |
|---|---|---|
| 無料枠の手厚さ | Cloudflare / Vercel が厚め、AWS / GCP / Azure は無料利用枠あり | 小さく始めるなら無料枠の厚いところが安心 |
| 公開までの手軽さ | Vercel・Cloudflare が速い、AWS / GCP / Azure は構成しだい | 「URL が出るまで」の早さに差が出る |
| できることの幅 | AWS / GCP / Azure が広い、Cloudflare は着実に拡大中、Vercel はフロント中心 | 大規模・多機能が必要なら総合クラウド |

順位はあくまで「初学者がとっつきやすいか」の目安です。規模が大きくなったり要件が特殊になれば、評価は
変わります。

### どんなときに Cloudflare を選ぶと良いか / 選ばないほうがいいか

選ぶと良い場面の例:

- **小さく速く公開したい**。AI で作ったアプリをまず世に出して反応を見たい
- フロント＋軽い API＋データベースという、**今回作ったような構成** を無料枠中心で動かしたい
- 画像や配布ファイルなど **読み出し（ダウンロード）が多い** データを扱う（R2 の egress 無料が効く）
- **サーバー管理をしたくない**。OS 更新やセキュリティパッチに時間を取られたくない

選ばないほうがいい場面の例:

- **重い常駐バックグラウンド処理** を動かしたい（ずっと起動して待つ・長時間まわす処理）
- 特定クラウドの **専用サービスに強く依存** している（例: そのクラウド固有のマネージドサービス前提）
- **既存の大きな Node / Express アプリをそのまま動かしたい**（移植コストが高くつくことがある）
- 細かく OS やミドルウェアまで **自分で握りたい**（この自由が欲しいなら VPS が向く）
