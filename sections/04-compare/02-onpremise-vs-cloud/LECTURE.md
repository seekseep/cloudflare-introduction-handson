# オンプレ vs クラウド

[比較の全体像](../01-overview/LECTURE.md) で見た 3 段階のうち、一番大きい第1段の問いがこれです。
**そもそもサーバー（ハードウェア）を自前で持つのか、それとも借りるのか。**

ここがいちばん上流の分かれ道です。ここで「借りる（クラウド）」を選べば、次の段（VPS vs SaaS）の話に
進めます。「自前で持つ（オンプレ）」を選ぶなら、そもそも検討する世界がまったく変わります。

結論から言うと、個人や小さく始める用途では、ほとんどの場合 **クラウド** で問題ありません。ただ
「なぜクラウドでいいのか」「どんなときにオンプレが要るのか」を理解しておくと、選択に納得感が出ます。

## 学ぶこと

- オンプレミス（自前のハード）とクラウド（借りるハード）の基本的な違い
- 初期費用（買い切り）と運用負荷の差、スケール（増減）への対応のしやすさ
- クラウドが初学者に向く理由と、それでもオンプレが選ばれる場面

## 説明

### 言葉の整理

- **オンプレミス（オンプレ）**：サーバーなどの機器を **自分（自社）で用意して持つ** 形。社内の
  サーバールームやデータセンターにハードを置き、自分たちで運用します。"on the premises"（敷地内に）が語源です。
- **クラウド**：サーバーを **借りて使う** 形。物理的な機器は事業者（Cloudflare / AWS など）が持っていて、
  利用者はその一部をネット越しに借ります。電気や水道のように「使う分だけ契約して使う」イメージです。

第1段の問いは、つきつめると「**ハードという資産を自分で抱えるか、抱えないか**」です。

### 3 つの観点で比べる

| 観点 | オンプレミス | クラウド |
|---|---|---|
| 初期費用 | 機器の購入で **まとまった出費**（買い切り） | ほぼ不要。**使った分だけ**支払い |
| 立ち上げの速さ | 機器の調達・設置に時間がかかる | アカウントを作れば **すぐ使える** |
| 運用・保守 | 故障対応・電源・空調まで **全部自分** | ハード障害は事業者側が面倒を見る |
| スケール（増減） | 増やすには機器を買い足す。**減らしにくい** | 必要に応じて **増減しやすい** |
| 向く規模 | 大規模・長期で安定した負荷 | 小さく始めて様子を見たい用途 |

### なぜ初学者にはクラウドが向くのか

一番大きいのは **初期費用と立ち上げの速さ** です。オンプレだと「まずサーバーを買う」ところから
始まりますが、クラウドなら **アカウントを作ればその日から** 公開を試せます。今回のハンズオンも、
ハードを 1 台も買わずに Cloudflare の無料枠だけでアプリを公開できました。これはクラウドだからです。

もう一つは **やめやすさ・変えやすさ** です。「作ってみたけど方向転換したい」というとき、クラウドなら
契約を止めるだけで済みます。買ってしまったハードはそうはいきません。小さく試して反応を見たい段階とは、
クラウドの「使った分だけ・いつでもやめられる」性質がとても相性がいいのです。

### それでもオンプレが選ばれる場面

クラウドが万能というわけではありません。オンプレが選ばれるのは、たとえば次のようなケースです。

- **規制・コンプライアンス**：データを社外（事業者の設備）に置けない、置きたくない業種・契約がある
- **既存資産がある**：すでに自前のデータセンターや機器があり、それを使い続けるほうが合理的
- **特殊な性能・物理要件**：特定のハードウェアや、極端に安定した大量処理が前提のシステム
- **長期で負荷が一定**：ずっと同じくらい使い続けるなら、買い切りのほうが総額で安くなることもある

:::notice
「クラウド＝安い」「オンプレ＝高い」と単純化はできません。**長期で負荷が一定** なら買い切りのオンプレが
総額で有利になることもあります。逆に **負荷が読めない・小さく始めたい** ならクラウドが有利です。
"使い方しだいで逆転する" という点だけ覚えておけば十分です。

### この段の結論

個人や小さく始める用途、そして今回のような「フロント＋軽い API＋DB」を世に出したいだけなら、
答えはほぼ **クラウド** です。上に挙げたオンプレの理由に当てはまらない限り、クラウドを選んで先に進んで
問題ありません。

クラウドを選んだら、次に問われるのは「**クラウドの中でも、サーバーを自分で管理するのか／管理を任せるのか**」
です。これが第2段の問いになります。
