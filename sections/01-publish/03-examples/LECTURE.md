# 静的ページアプリの例

前の章では「ひとことボード」のフロントを Pages に公開しました。あれは HTML と JavaScript だけの
**静的ページ** でしたが、実は「見た目だけ」に見えて、意外といろいろなアプリが静的ページのまま作れます。

このレクチャーでは、サーバーを用意しなくても作れるアプリの例を **3 つ** 用意しました。まずはローカルで
動かして眺めてみて、「フロントだけでどこまでできるか」の感覚を掴みましょう。

## 学ぶこと

- **静的ページアプリ** とは何か（ブラウザの中だけで計算・保存・再生が完結するアプリ）
- サーバーを持たない静的ページは、**公開のハードルもセキュリティ上のリスクも低い** こと
- 逆に「みんなで共有する保存」など、フロントだけでは作れないものは何か

## 説明

### サーバーがなくても、これくらいは作れる

Web アプリと聞くと「サーバーが要る」と思いがちですが、**ユーザーの操作にその場で反応するだけ** の
アプリなら、サーバーなしで作れます。今回のサンプルは次の 3 つです。

| アプリ | URL | 何をしている | 保存 |
|---|---|---|---|
| ⚖️ BMI 計算機 | `/bmi-calculator` | 入力値から計算して表示するだけ | なし |
| 📒 単語帳 | `/word-list` | 1問4択のクイズで単語を覚える（難易度3段階） | ベストスコアをブラウザ内（localStorage） |
| 🎹 音のなるおもちゃ | `/toy-sound` | ボタンで音を鳴らす | なし |

トップページ（`/`）はこの 3 つへのメニューになっています。

いずれも **入力した内容を外のサーバーへ送りません**。計算はその場で、保存が要るもの（単語帳）も
あなたのブラウザの中（localStorage）に書くだけです。

> 🎹 の音は、音声ファイルではなく **ブラウザ内蔵の Web Audio API** でその場で合成しています。
> 録音済みの効果音に差し替えたい場合の無料音源サイト（CC0・表記不要）は、`/toy-sound` ページ内に
> リンクを載せてあります。

### TODO 1: ローカルで動かす

このレクチャーのコマンドは、すべて **`sections/01-publish/03-examples/` フォルダの中** で実行します。
リポジトリのルートからこのフォルダへ移動し、依存をインストールします。

```bash
cd sections/01-publish/03-examples
npm install
```

ローカルプレビューを起動します（`02-pages` と同じ `wrangler pages dev`）。

```bash
npx wrangler pages dev ./public
```

表示された `http://localhost:8788`（ポートは変わることがあります）をブラウザで開くと、3 つのアプリへの
メニューが出ます。それぞれ開いて触ってみましょう。

- **⚖️ BMI 計算機** … 身長・体重を入れて「計算する」。バックエンドは一切呼ばれません
- **📒 単語帳** … 難易度（初級・中級・上級）を選び、1問4択のクイズに答えて単語を覚えます。ベストスコアが保存され、別のブラウザや端末では見えないことも確認
- **🎹 音のなるおもちゃ** … ボタン、またはキーボードの `A`〜`K` で音が鳴ります

### TODO 2: Cloudflare Pages に公開する（任意）

`02-pages` とまったく同じ手順で公開できます。公開フォルダ `./public` を指定して実行し、初回は対話で
プロジェクト名を入力します（例: `static-examples-あなたの名前`）。

```bash
npx wrangler pages deploy ./public
```

公開 URL（`https://<name>.pages.dev`）が表示されたら、`https://<name>.pages.dev/bmi-calculator` の
ように各アプリにもアクセスできます。スマホからも開いてみましょう。

> フォルダ分け（`public/bmi-calculator/index.html` など）がそのまま URL（`/bmi-calculator`）になります。
> Pages は「フォルダの構造をそのまま URL として配信する」だけ、というのがよく分かります。

不要になったら [02-pages のレクチャー](../02-pages/LECTURE.md#todo-4-公開したものを削除する) と同じ手順で
削除できます。

## コラム

### なぜ「フロントだけ」は安全側なのか

サーバー（API やデータベース）を持つと、外から届いたリクエストを受け取って処理する「入口」ができます。
入口があるということは、**攻撃の対象になる場所ができる** ということでもあります。入力の検証漏れ、認証の
不備、データベースへの不正な問い合わせ（SQL インジェクション）など、気をつけることが一気に増えます。

一方、今回の 3 つのように **フロントだけで完結する静的ページ** は、そもそも受け取って処理するサーバーが
ありません。ユーザーが入れた値はその人のブラウザの中で消費されるだけで、外に保存も送信もされない。
だから **公開のハードルも、セキュリティ上のリスクも低い** のです。

もちろん「フロントだけ」では、みんなで共有する掲示板や、ログインの必要なサービスは作れません。そういう
アプリを作るときにサーバーが必要になり、同時にセキュリティも考える——という順序を、このハンズオンでは
このあと少しずつ体験していきます。

### 静的ページアプリに向いているもの・向かないもの

- **向いているもの**: 計算ツール（BMI・割り勘・単位変換）、その場で完結するゲーム、入力フォームの
  下書き、このおもちゃのような表現物。データを自分の端末にだけ持てば十分なもの
- **向かないもの**: 複数人で共有する掲示板、ログインが要るサービス、他人に見せる投稿の保存。これらは
  次のセクション以降で **API（Workers）とデータベース（D1）** を足して実現します
