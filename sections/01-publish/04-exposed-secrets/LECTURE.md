# 静的ページの落とし穴（フロントに秘密を置かない）

前の章では「フロントだけの静的ページは、公開のハードルもセキュリティ上のリスクも低い」と説明しました。
それはその通りなのですが、**ひとつだけ大きな落とし穴** があります。それは

> **「画面に出していない」＝「隠せている」ではない**

ということです。フロント（HTML・JavaScript）に書いたものは、見た目に出す・出さないに関係なく、
**まるごとユーザーのブラウザに届いています**。このレクチャーでは、それを一発で体感できる
「やってはいけない例」を用意しました。

## 学ぶこと

- フロントに置いた HTML / JavaScript は、**表示していなくても全部ユーザーに渡っている** こと
- 「画面でマスク・非表示にする」だけでは **秘密を守れない** こと（F12 で丸見え）
- 顧客情報・API キー・パスワードのような秘密は **サーバー側に置く** べきで、それが次のセクションで
  Workers・D1 を学ぶ動機になること

## 説明

### このデモがやっている「まずいこと」

デモの会員リストは、画面上ではメールアドレスと電話番号を <code>●●●</code> でマスクして表示しています。
一見「個人情報は隠せている」ように見えます。

でも、その元データは JavaScript ファイル（`public/main.js`）の中に **そのまま** 書かれていて、
このファイルはブラウザにまるごとダウンロードされています。マスクは「画面に描くときだけ」の
見せかけで、データ自体は隠せていません。

### TODO 2: F12 で「隠したはずの情報」を見る

ブラウザで **F12** キー（または右クリック → 「検証」／「Inspect」）を押して開発者ツールを開きます。
次の 3 つのどれでも、隠したはずの個人情報が確認できます。

1. **Console（コンソール）で見る** ― 一番かんたん
   コンソールのタブを開き、次のように打って Enter を押します。

   ```js
   customers
   ```

   マスクする前のメールアドレス・電話番号・住所・生年月日・社内メモまで、配列がまるごと表示されます。
   さらにこう打てば、CSV のように並べて見ることもできます。

   ```js
   console.table(customers)
   ```

2. **Sources（ソース）で見る**
   「Sources」タブから `main.js` を開くと、ソースコードの中に個人情報が直書きされているのが読めます。

3. **Network（ネットワーク）で見る**
   「Network」タブを開いた状態でページを再読み込みし、`main.js` を選んで「Response（応答）」を見ると、
   サーバーから届いたファイルの中身として個人情報が入っているのが分かります。

つまり、**画面のマスクはまったく防御になっていません**。ブラウザに送った時点で、開発者ツールを開ける人
なら誰でも読めます。

:::danger
これは **絶対にやってはいけない例** です。顧客の個人情報・API キー・パスワード・秘密トークンなどを、
フロントの HTML / JavaScript / 設定ファイルに書いてはいけません。「画面に出さない」「変数名を分かりにくく
する」「マスクして表示する」では守れません。ブラウザに渡した時点で公開したのと同じです。
:::

### どうすればよかったのか

守りたいデータ（顧客情報など）は **サーバー側** に置き、ブラウザには **必要な分だけ** を返すのが基本です。

- 会員一覧を表示する画面なら、**サーバー（API）が「その人が見てよい範囲」だけを返す**
- そもそもマスクが必要な項目（フルのメール・電話）は、**必要になるまでブラウザに送らない**
- ログインした本人・権限のある担当者かをサーバー側で確認してから返す

このデモには「サーバー」がありません。だから全データをフロントに置くしかなく、結果として丸見えに
なりました。**共有・保存・秘密の保持が必要になった瞬間に、サーバーが必要になる** ――前の章の最後で
触れた話が、ここでつながります。

## コラム

### 「難読化」は防御ではない

「変数名を `a` `b` にする」「コードを圧縮（minify）して読みにくくする」といった **難読化** をしても、
データそのものはブラウザに届いています。時間をかければ復元できますし、そもそも Network タブで元の
レスポンスを見れば一発です。難読化は「うっかり見えにくくする」程度で、**秘密を守る手段にはなりません**。

### よくある実際の事故

- フロントの JavaScript に **API キーやシークレットトークン** を直書きし、公開後に第三者に使われて
  課金が発生する
- 「管理者だけに見せる」つもりの項目を、画面では隠しつつ **データはフロントに全部送っており**、
  DevTools から抜かれる
- `.env` に書いたつもりの値を、ビルド時にフロント側のバンドルへ埋め込んでしまい公開される

いずれも「サーバー側に置くべきものを、フロントに置いた」ことが原因です。
