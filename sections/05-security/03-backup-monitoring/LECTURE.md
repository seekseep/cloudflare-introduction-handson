# バックアップ・監視・アラート

アプリを公開したら、それで終わりではありません。**動き続けている間に何かが起きる**前提で、運用の
仕組みを用意しておく必要があります。

データが消えたとき復旧できるか（バックアップ）、おかしな状態に気づけるか（監視）、そして自分が見て
いないときでも異常を知らせてくれるか（アラート）。この3つは、個人開発であっても最初に押さえておきたい
土台です。

「ひとことボード」の投稿は D1（データベース）に入っています。この章では、そのデータをどう守り、
どう異常に気づくかを整理します。

## 学ぶこと

- `wrangler d1 export` で D1 を SQL ダンプとして取得できる
- バックアップは「取れている」だけでなく「戻せる」ことが重要
- Workers の observability、ダッシュボードのログ／メトリクス、`wrangler tail`
- Cloudflare の Notifications でエラー率・リクエスト急増・課金しきい値を通知できる
- 「異常に気づける仕組み」を最初に作るという考え方

## 説明

### バックアップ：D1 のデータを守る

データベースのデータは、操作ミスやバグ、想定外の事態で失われることがあります。投稿データが消えて
しまったら取り返しがつきません。だから定期的にバックアップを取ります。

D1 は `wrangler d1 export` でデータベースの中身を SQL ファイル（ダンプ）として書き出せます。

```bash
# 本番 D1 を SQL ファイルに書き出す
npx wrangler d1 export hitokoto-db --remote --output ./backup.sql
```

書き出した `backup.sql` を、手元や別のストレージに保管します。Cloudflare の **R2**（オブジェクト
ストレージ）に置いておくのも選択肢です。手元の PC だけに置くより、別の場所にもコピーがあるほうが安全です。

理想は、これを**定期的に**（手動でも、後で自動化してもよい）実行する運用です。

#### 大事なのは「戻せる」こと

ありがちな落とし穴は、「バックアップを取っているつもり」で、いざというときに**戻せない**ことです。
ファイルが壊れていた、書き出していたつもりで失敗していた、復元手順を誰も知らない——これでは意味が
ありません。

一度は実際に、ダンプから別のデータベース（ローカルやテスト用）へ復元してみて、**ちゃんと戻ることを
確認**しておきましょう。バックアップは「取得」と「復元の確認」がセットです。

### 監視：状態を見えるようにする

問題に対処するには、まず「何が起きているか」が見える必要があります。Workers にはそのための仕組みが
あります。

#### observability を有効にする

`wrangler.jsonc` で observability を有効にすると、Worker のログやメトリクスが収集され、ダッシュ
ボードから確認できるようになります。

```jsonc
{
  "observability": {
    "enabled": true
  }
}
```

#### ダッシュボードのログ／メトリクス

Cloudflare のダッシュボードでは、リクエスト数、エラーの発生状況、CPU 時間などのメトリクスや、
ログを確認できます。「最近エラーが増えていないか」「アクセスが急増していないか」を、ここで定期的に
見る習慣をつけます。

#### wrangler tail でライブログ

今まさに動いている Worker のログをリアルタイムで見たいときは `wrangler tail` が便利です。

```bash
npx wrangler tail
```

実際にリクエストを送りながらログを眺めると、不具合の原因調査がはかどります。

### アラート：見ていなくても気づく

監視は「自分が見に行ったとき」しか役に立ちません。問題は、あなたが寝ている間や忙しいときにも起きます。
そこで、**異常を自動で知らせてくれる仕組み**＝アラートを用意します。

Cloudflare の **Notifications** では、たとえば次のような条件で通知（メールなど）を設定できます。

- エラー率が一定を超えたとき
- リクエストが急増したとき
- 課金（使用量）がしきい値に達したとき

特に課金しきい値の通知は、前章で触れた**財布攻撃や想定外の使用量**にいち早く気づくために重要です。
※設定できる項目や条件はプランや時期で変わるため、最新は公式ドキュメントを確認してください。

大切なのは考え方です。**「異常に気づける仕組みを最初に作っておく」**こと。問題が起きてから慌てて
監視を入れるのではなく、公開と同時にアラートを仕込んでおくと、被害を小さく抑えられます。

### まず最低限やること

完璧を目指さず、まずは次の3つから始めれば十分です。

![運用の土台は「バックアップ」「監視」「アラート」の3本柱でできている](./images/02-three-pillars.svg)

1. **バックアップ**：`wrangler d1 export` でデータを書き出し、一度は復元できることを確認する
2. **監視**：`observability` を有効にし、ダッシュボードを時々見る習慣をつける
3. **アラート**：課金しきい値とエラー率の通知を Notifications で設定する
