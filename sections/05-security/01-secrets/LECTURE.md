# 秘匿情報（APIキー等）の扱い方

アプリを公開して運用しはじめると、外部サービスの API キーやトークン、データベースのパスワードなど、
人に見せてはいけない「秘匿情報（シークレット）」を扱う場面が出てきます。

これらの扱いを一度でも間違えると、**他人にあなたの権限で外部サービスを使われ、不正利用や高額請求に
つながります**。しかも一度インターネットや Git に出てしまった値は「なかったこと」にできません。

この章では、Cloudflare のアプリ（「ひとことボード」）を例に、シークレットを安全に持つための基本ルールを
読み物として整理します。コードは書きませんが、考え方と実際のコマンドはこの章で身につきます。

## 学ぶこと

- シークレットを漏らすと何が起きるか（不正利用・高額請求・取り消せない）
- Cloudflare では「暗号化シークレット」と「平文の設定値（vars）」を使い分ける
- ローカルは `.dev.vars`、本番は `wrangler secret put` で渡す
- シークレットは必ず **サーバー（Worker）側** に置く。フロントに書くと丸見え
- `wrangler secret put` は値を引数で渡さず対話入力する（履歴に残さないため）

## 説明

### なぜコードや Git に入れてはいけないのか

API キーやトークンは「あなたであることの証明書」です。これを持っている人は、あなたとして外部サービスを
呼び出せます。だからこそ、次のような置き場所はすべて危険です。

- ソースコードに直接書く（`const apiKey = "sk-xxxx"`）
- Git にコミットする（公開リポジトリはもちろん、非公開でも共有範囲が広がるほど危険）
- フロントエンド（`index.html` や公開される JS）に書く

特に Git は、**一度コミットすると履歴に残り続けます**。後のコミットで消しても、過去の履歴をたどれば
読めてしまいます。公開リポジトリに上げてしまった鍵は「すでに漏れたもの」として、発行元で**無効化（ローテーション）**
するのが基本対応です。

漏洩した鍵が悪用されると、たとえば従量課金の外部 API を大量に叩かれて高額請求が来る、データを盗まれる、
といった被害につながります。

### 暗号化シークレットと公開してよい設定値

Cloudflare Workers では、Worker に値を渡す方法が大きく2つあります。秘密にすべきかどうかで使い分けます。

| 観点 | 暗号化シークレット | 平文の設定値（vars） |
| --- | --- | --- |
| 設定方法 | `npx wrangler secret put NAME`（対話入力） | `wrangler.jsonc` の `vars` に記述 |
| 値の見え方 | ダッシュボードでも値は表示されない（書き換えのみ） | 設定ファイルに平書きされ、Git にも残る |
| 向いている用途 | API キー、トークン、DB パスワードなど | 公開してよい設定（機能フラグ、公開 URL、表示名など） |
| Git に入れてよいか | 値は入れない（コマンドで投入） | 入れてよい（秘密ではない前提） |

`vars` は「設定ファイルに書いてあること自体が問題にならない値」だけに使います。少しでも秘密にしたい値は
必ず暗号化シークレットにします。

`wrangler.jsonc` での `vars` の例（これは公開してよい値だけ）:

```jsonc
{
  "vars": {
    "APP_NAME": "ひとことボード",
    "PUBLIC_SITE_URL": "https://example.pages.dev"
  }
}
```

本番のシークレットはコマンドで投入します。

```bash
npx wrangler secret put EXTERNAL_API_KEY
# 実行するとプロンプトが出るので、そこに値を貼り付ける
```

このとき、**値をコマンドの引数に書かないでください**。引数に書くとシェルの履歴（`~/.zsh_history` 等）に
平文で残ってしまいます。`wrangler secret put NAME` は対話入力（プロンプト）で受け取る設計なので、
そのまま貼り付けるのが安全です。

### ローカル開発でのシークレット（.dev.vars）

ローカルで `wrangler dev` を動かすときは、本番のシークレットを使わず、ローカル用の値を `.dev.vars`
ファイルに書きます。形式は `.env` と同じ `KEY=VALUE` です。

```bash
# .dev.vars（このファイルは Git に入れない）
EXTERNAL_API_KEY=local-development-key
```

ここで最重要なのは、**`.dev.vars` を絶対に Git に入れないこと**です。`.gitignore` に次の行が入って
いることを確認します。

```text
.dev.vars
.wrangler/
```

このリポジトリの `.gitignore` には、すでに `.dev.vars`（および `.dev.vars.*`）と `.wrangler/` が
登録されています。`.wrangler/` にはローカルの状態やキャッシュが入るため、こちらも除外しておきます。

### Worker コードからの参照

シークレットも `vars` も、Worker のコードからは **環境（env）経由** で同じように参照します。コードに
値を直書きせず、名前で取り出すのがポイントです。

- 素の Workers なら `env.EXTERNAL_API_KEY`
- Hono を使っているなら `c.env.EXTERNAL_API_KEY`

こうしておけば、ローカルでは `.dev.vars` の値、本番では `wrangler secret put` で入れた値が、コードを
変えずに自動で切り替わります。

### やりがちな失敗：フロントにシークレットを書く

一番多い事故が、**公開されるフロント側にシークレットを書いてしまう**ことです。

`index.html` やそこから読み込まれる JavaScript は、ブラウザに丸ごとダウンロードされます。つまり
**誰でも開発者ツールやソース表示で中身を読めます**。ここに API キーを書くと、世界中に公開したのと
同じです。

正しくは、シークレットを使った外部 API 呼び出しは **必ずサーバー（Worker）側で行い**、フロントは自分の
Worker（API）だけを呼ぶ構成にします。「ひとことボード」でいえば、フロントは `/api/...` を叩くだけで、
外部サービスの鍵は Worker の中だけに留めます。

### まず最低限やること

- API キー・トークン・パスワードは `wrangler secret put` で入れる（`vars` には書かない）
- ローカル用の値は `.dev.vars` に書き、`.gitignore` で除外する
- フロントにシークレットを書かない。外部 API は Worker から呼ぶ
- 万一 Git に上げてしまったら、その鍵は無効化して作り直す
